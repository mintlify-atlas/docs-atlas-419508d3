## Overview

The Mini-Compilador Educativo transforms source code through six sequential phases, each building on the output of the previous one. This page provides an in-depth look at how a sample program flows through the entire compilation pipeline.

## Example Program

We'll trace this simple program through all phases:

```javascript
let a = 5;
let b = 10;
let c = a + b * 2;
print c;
```

## Phase 2: Syntactic Analysis (Parser)

<Steps>
  <Step title="Input">
    Token stream from Scanner
  </Step>
  
  <Step title="Processing">
    The `Parser` uses **recursive descent** to match tokens against grammar rules:
    
    1. `parsear()` в†’ processes statements until EOF
    2. `_sentencia()` в†’ dispatches to `_declaracion_variable()` or `_sentencia_print()`
    3. `_expresion()` в†’ delegates to `_suma_resta()` в†’ `_multiplicacion_division()` в†’ `_primario()`
    4. Operator precedence handled by nesting depth
  </Step>
  
  <Step title="Output">
    Abstract Syntax Tree (AST)
  </Step>
</Steps>

### Precedence in Action

<Info>
**Expression:** `a + b * 2`

**Why it parses as** `a + (b * 2)` **and not** `(a + b) * 2`:

1. `_expresion()` calls `_suma_resta()`
2. `_suma_resta()` calls `_multiplicacion_division()` for left side в†’ gets `a`
3. Sees `+`, so calls `_multiplicacion_division()` for right side
4. `_multiplicacion_division()` processes `b * 2` completely (consuming both tokens) before returning
5. Result: `ExpresionBinaria(+, a, ExpresionBinaria(*, b, 2))`

**Source:** `compfinal.py:802-843`
</Info>

## Phase 3: Semantic Analysis

<Steps>
  <Step title="Input">
    AST from Parser
  </Step>
  
  <Step title="Processing">
    The `AnalizadorSemantico` traverses the AST with these checks:
    
    **Variable Tracking:**
    - Maintains `variables_declaradas` set
    - On `DeclaracionVariable`: analyze expression first, then add variable
    - On `Identificador`: verify variable exists in set
    
    **Division by Zero:**
    - On `ExpresionBinaria` with `/` operator
    - If right side is `NumeroLiteral` with `valor == 0`, report error
  </Step>
  
  <Step title="Output">
    Boolean success flag + error list
  </Step>
</Steps>

## Phase 4: Intermediate Code Generation

<Steps>
  <Step title="Input">
    Validated AST
  </Step>
  
  <Step title="Processing">
    The `GeneradorIR` creates **Three Address Code (TAC)**:
    
    - Each operation becomes one TAC instruction
    - Temporary variables (`t0`, `t1`, ...) store intermediate results
    - Variables use their original names
  </Step>
  
  <Step title="Output">
    List of TAC instructions as strings
  </Step>
</Steps>

### Generated IR Code

<CodeGroup>
```python Three Address Code
a = 5
b = 10
t0 = b * 2
t1 = a + t0
c = t1
print c
```

```python Generation Logic
# For: let c = a + b * 2;

1. generar_expr(ExpresionBinaria(+, ...))
   
2. Process left side:
   - generar_expr(Identificador('a'))
   - Returns: "a"
   
3. Process right side:
   - generar_expr(ExpresionBinaria(*, ...))
     - Left: generar_expr(Identificador('b')) в†’ "b"
     - Right: generar_expr(NumeroLiteral(2)) в†’ 2
     - Create temp: t0 = nueva_temp()
     - Emit: "t0 = b * 2"
     - Return: "t0"
   
4. Create temp: t1 = nueva_temp()
5. Emit: "t1 = a + t0"
6. Return: "t1"

7. generar_sentencia(DeclaracionVariable):
   - Emit: "c = t1"
```
</CodeGroup>

### TAC Properties

<CardGroup cols={2}>
  <Card title="Explicit Ordering" icon="list-ol">
    Operations occur in sequential order, making control flow obvious.
  </Card>
  
  <Card title="Single Operator" icon="code">
    Each instruction has at most one operator, simplifying optimization and code generation.
  </Card>
  
  <Card title="Platform Independent" icon="globe">
    TAC is abstract; not tied to any specific CPU architecture.
  </Card>
  
  <Card title="Optimization Ready" icon="rocket">
    Easy to apply optimizations like constant folding, dead code elimination, etc.
  </Card>
</CardGroup>

<Steps>

## Phase 6: Assembly Code Generation

<Steps>
  <Step title="Input">
    AST
  </Step>
  
  <Step title="Processing">
    The `GeneradorASM` produces x86 assembly for EMU8086:
    
    1. **Data section:** Declare all variables as words (`dw`)
    2. **Code section:** Translate each statement:
       - Use `AX` register as accumulator
       - Use stack for nested expressions
       - Call `print_num` procedure for output
    3. **Utility procedures:** Include `print_num` for integer output
  </Step>
  
  <Step title="Output">
    Complete `.asm` file as string
  </Step>
</Steps>

```asm Expression Detail: a + b * 2
; Evaluate: a + (b * 2)

; Step 1: Load left operand (a)
mov ax, a          ; AX = value of a
push ax            ; Save a on stack

; Step 2: Evaluate right side (b * 2)
mov ax, b          ; AX = value of b
push ax            ; Save b on stack
mov ax, 2          ; AX = 2
mov bx, ax         ; BX = 2
pop ax             ; AX = b (restore)
imul bx            ; AX = b * 2

; Step 3: Perform addition
mov bx, ax         ; BX = b * 2
pop ax             ; AX = a (restore)
add ax, bx         ; AX = a + (b * 2)

; Step 4: Store result
mov c, ax          ; c = result
```
</CodeGroup>

### Assembly Generation Strategy

<Tabs>
  <Tab title="Stack-Based Evaluation">
    Binary expressions use stack to preserve left operand:
    
    ```
    1. Evaluate left в†’ result in AX
    2. Push AX
    3. Evaluate right в†’ result in AX
    4. Move AX to BX
    5. Pop AX (restore left)
    6. Perform operation (AX op BX в†’ AX)
    ```
    
    This handles nested expressions naturally through recursion.
  </Tab>
  
  <Tab title="Register Usage">
    - **AX:** Primary accumulator, holds expression results
    - **BX:** Right operand for binary operations
    - **CX:** Loop counter in `print_num`
    - **DX:** DOS interrupt parameter, digit extraction
    - **DS:** Data segment register
    - **Stack:** Temporary storage for nested expressions
  </Tab>
  
  <Tab title="DOS Interrupts">
    Uses INT 21h for I/O:
    
    - **AH=02h:** Print single character (DL = ASCII code)
    - **AH=09h:** Print string (DX = string address, '$' terminated)
    - **AH=4Ch:** Exit program
  </Tab>
  
  <Tab title="Print Number Logic">
    Converts integer in AX to decimal string:
    
    1. Divide by 10 repeatedly
    2. Push remainders (digits) onto stack
    3. Pop and print each digit (in correct order)
    
    Handles zero as special case.
  </Tab>
</Tabs>

## Error Handling Across Phases

<AccordionGroup>
  <Accordion title="Lexical Errors (Phase 1)">
    **Detected by:** `Scanner`  
    **Examples:**
    - Invalid characters: `let x = 5@;`
    - Unexpected symbols: `let x = $100;`
    
    **Handling:**
    - Error added to `Scanner.errores`
    - Token type set to `ERROR`
    - Scanning continues to find more errors
    
    **Output:**
    ```
    Error lГ©xico en lГ­nea 1, columna 10: carГЎcter inesperado '@'
    ```
  </Accordion>
  
  <Accordion title="Syntax Errors (Phase 2)">
    **Detected by:** `Parser`  
    **Examples:**
    - Missing semicolon: `let x = 5`
    - Wrong token order: `= x 5 let;`
    - Unmatched parentheses: `let x = (5 + 3;`
    
    **Handling:**
    - `ErrorSintaxis` exception raised
    - Error caught and added to `Parser.errores`
    - Synchronization to next statement boundary
    - Parsing continues
    
    **Output:**
    ```
    Error de sintaxis en lГ­nea 1, columna 10: Se esperaba ';' al final de la declaraciГіn. Se encontrГі ''
    ```
  </Accordion>
  
  <Accordion title="Semantic Errors (Phase 3)">
    **Detected by:** `AnalizadorSemantico`  
    **Examples:**
    - Undeclared variable: `print x;` (without `let x = ...;`)
    - Division by zero: `let x = 10 / 0;`
    
    **Handling:**
    - Error added to `AnalizadorSemantico.errores`
    - Analysis continues to find more errors
    
    **Output:**
    ```
    Error semГЎntico en lГ­nea 2, columna 7: la variable 'x' no ha sido declarada
    ```
  </Accordion>
  
  <Accordion title="No Runtime Errors (Phases 4-6)">
    Once semantic analysis passes, subsequent phases assume correctness:
    
    - **IR Generation:** Always succeeds with valid AST
    - **Interpretation:** May encounter runtime division by zero (not caught)
    - **Assembly Generation:** Produces syntactically correct code
    
    <Warning>
    The compiler does not detect:
    - Runtime division by zero (variable divisor)
    - Integer overflow
    - Stack overflow from deep recursion
    </Warning>
  </Accordion>
</AccordionGroup>

## Performance Characteristics

<CardGroup cols={2}>
  <Card title="Time Complexity" icon="clock">
    - **Lexer:** O(n) where n = character count
    - **Parser:** O(n) where n = token count
    - **Semantic:** O(n) where n = AST node count
    - **IR/ASM:** O(n) where n = AST node count
    - **Overall:** O(n) linear in input size
  </Card>
  
  <Card title="Space Complexity" icon="memory">
    - **Token list:** O(n) tokens
    - **AST:** O(n) nodes
    - **Symbol table:** O(v) variables
    - **IR code:** O(n) instructions
    - **Overall:** O(n) linear in input size
  </Card>
</CardGroup>

<Check>
The compiler completes in milliseconds for typical programs (< 100 lines). GUI shows compilation time in status bar.
